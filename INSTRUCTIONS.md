# Operations and troubleshooting

For first installation, start with [README.md](README.md). This document covers
updates, operating details and diagnosis after the adapter has been installed.

## Supported topology

```text
Community gateway Pi
  BlueZ + gateway for SANlight Mesh + local Mosquitto :1883
                     ^
                     | trusted LAN
                     |
ioBroker host
  ioBroker.sanlightmesh instance
```

The broker normally runs on the community gateway Pi. Use `localhost` only when
ioBroker and the complete gateway are deliberately installed on the same Pi.
Port `1883` is authenticated but not encrypted; never expose it to the internet.

## Credentials

The gateway installer prints the ioBroker username and the command for reading
the protected password:

```bash
sudo cat /etc/sanlight-mesh-mqtt-gateway/iobroker-mqtt-password.txt
```

Do not paste the password into logs, issues or support chats. Delete temporary
notes after entering it into ioBroker.

## Configuration reference

Required installation-specific values:

| Setting           | Value                                            |
| ----------------- | ------------------------------------------------ |
| Gateway host      | gateway Pi IP/hostname, or `localhost` on one Pi |
| Username/password | generated by the gateway installer               |
| Gateway ID        | exact gateway installer value                    |

Normal defaults:

| Setting             | Default                                       |
| ------------------- | --------------------------------------------- |
| MQTT port           | `1883`                                        |
| TLS                 | disabled for the documented trusted-LAN setup |
| Topic prefix        | `sanlightmesh/v1`                             |
| Command TTL         | `30` seconds                                  |
| Brightness debounce | `1000` ms                                     |
| Explicit blackout   | disabled                                      |

The gateway ID must match the broker ACL and topic root generated by the gateway
installer.

## Health states

`info.connection` becomes true only when all three component states are true:

- `info.mqttConnected` — MQTT transport connected;
- `info.gatewayOnline` — retained gateway availability is `online`;
- `info.protocolCompatible` — MQTT API v1 and the configured gateway ID match.

`info.lastError` contains the most recent adapter-level configuration, MQTT or
message-validation error.

## Read-only smoke test

Trigger:

```text
lamps.<address>.control.refresh
```

The state must be written with `ack=false`; ioBroker UI controls normally do
this automatically. A successful flow produces:

```text
lamps.<address>.command.lastStatus = verified
lamps.<address>.state.available = true
lamps.<address>.state.verified = true
```

This is the preferred first diagnostic because it does not change brightness.

## Brightness writes

`lamps.<address>.control.maxBrightness` accepts only `20..100`.

The requested control state and verified lamp report are deliberately separate.
Watch:

```text
lamps.<address>.command.pending
lamps.<address>.command.lastStatus
lamps.<address>.command.lastError
lamps.<address>.state.maxBrightness
```

A successful command ends with `lastStatus=verified` and an updated verified
state. The gateway enforces additional rate limiting; do not create rapid or
per-second automation loops.

### Current effective brightness

Use `lamps.<address>.state.liveBrightnessPercentEstimate` as the readable current
output indicator. It keeps one decimal place: the validated example `33.4%` is
displayed as `34%` by the SANlight app because the app rounds to whole percent.
Check `liveVerified` and `liveVerifiedAt` before using the value in automation.

The former `state.liveBrightnessRaw` ioBroker object is no longer created and is
removed automatically when the adapter next discovers the lamp. The raw field
still travels inside MQTT API v1 and is validated internally, so this cleanup is
not a protocol break.

## Lamp clock operations

Observed values are snapshots:

```text
lamps.<address>.state.lampClockSeconds
lamps.<address>.state.lampClock
gateway.info.localClockSeconds
gateway.info.localClock
```

Use `gateway.control.refreshInfo` for a fresh gateway-local reference without a
Bluetooth Mesh operation. Use `lamps.<address>.control.refresh` when a fresh lamp
clock is required; this performs the normal lamp read.

To copy gateway local time to a lamp, trigger `control.syncClockNow`. To set an
arbitrary real or virtual lamp time, write `clockTargetSeconds` (`0..86399`) or
`clockTargetTime` (`HH:MM` or `HH:MM:SS`), then trigger the matching apply button.
`24:00` is rejected. Editing target inputs alone does not write to the lamp.

Clock writes still use gateway queueing, readback and verification. All-lamp
results can be partial and are reflected in the affected lamp command states.
The displayed clock values are snapshots. The adapter changes lamp clocks only
after an explicit sync or apply action.

## Blackout

Blackout is disabled by default. Enable it only after confirming that the
connected dimmer type supports the explicit 0% workflow and after reading the
gateway documentation. Do not use blackout as a connectivity test.

## Multiple instances

Use one instance per physical gateway:

```text
sanlightmesh.0 -> gateway A address, credentials and gateway ID
sanlightmesh.1 -> gateway B address, credentials and gateway ID
```

Each instance has an independent object namespace and exact MQTT subscriptions.

## Updating from GitHub

The adapter is not yet part of an ioBroker repository, so the normal
`iobroker upgrade sanlightmesh` command cannot update it.

Preferred method in ioBroker Admin:

1. Open **Adapters**.
2. Choose **Install from custom URL**.
3. In the **Any** / custom URL field, enter only:

    ```text
    https://github.com/Nibbels/ioBroker.sanlightmesh
    ```

4. Install it again over the existing adapter.

The command-line equivalent is:

```bash
iobroker url https://github.com/Nibbels/ioBroker.sanlightmesh sanlightmesh --debug
```

Do not use a command of this form:

```text
iobroker upgrade sanlightmesh@https://github.com/Nibbels/ioBroker.sanlightmesh
```

`upgrade` expects an adapter version from a configured ioBroker repository; the
GitHub repository is a custom installation source and must be handled by the
`url` command instead.

Existing instances and their native configuration are retained. After updating,
confirm that the instance is enabled and check the four health states.

### Tagged release installation

After the `v0.2.0` tag exists, install that immutable source revision with:

```bash
iobroker url Nibbels/ioBroker.sanlightmesh#v0.2.0 sanlightmesh --debug
```

Use the untagged GitHub URL only when intentionally testing the current `main`
branch.

## Troubleshooting

### Adapter remains red or restarts

Check the ioBroker log for `sanlightmesh` messages. A runtime parse error before
MQTT connection is an adapter compatibility problem, not a broker-password
problem.

Collect a redacted log with the ioBroker and Node.js versions. Never include
passwords or private Mesh data. Report reproducible adapter defects through
[GitHub Issues](https://github.com/Nibbels/ioBroker.sanlightmesh/issues).

### MQTT is not connected

Verify:

- the gateway host points to the gateway Pi, or `localhost` on one Pi;
- TCP port `1883` is reachable on the trusted LAN;
- username and password belong to that exact gateway installation;
- TLS is disabled for the default setup;
- another MQTT client is not repeatedly changing or locking the same setup.

### MQTT connected but gateway offline

On the gateway Pi:

```bash
sudo sanlight-gateway doctor
sudo sanlight-gateway status
sudo sanlight-gateway logs
```

Do not request Mesh keys or private state files for diagnosis.

### Protocol compatible is false

Confirm that the configured gateway ID matches the installed gateway ID and
that both projects use MQTT API v1. Do not widen subscriptions to discover a
working gateway automatically.

### Lamp objects are missing

Check gateway availability and protocol compatibility first, then trigger a
read-only refresh. Nodes missing from a newer topology are marked unavailable
rather than deleted, preserving scripts, aliases and history settings.

## Safe diagnostic information

Safe to share after review:

- adapter version and instance number;
- ioBroker and Node.js versions;
- broker hostname/IP with the password omitted;
- configured gateway ID and topic prefix;
- `info.*`, `gateway.info.*` and lamp state objects;
- redacted adapter logs;
- redacted output from `sanlight-gateway collect-diagnostics`.

Never share MQTT passwords, the private CDB, Mesh keys, `.state/` contents or
BlueZ tokens.

## Development checks

Prerequisites: Node.js 20 or newer and npm.

```bash
npm ci
npm test
npm run format:check
```

The adapter is installed from `src/main.ts`; generated `build/` and `build-test/`
files are not committed.
